# Rule 17: AI自动化流程的可观测性是强制约束

> Created By [RV](mailto:rodney.vin@gmail.com), and licensed with Creative Commons "[CC BY-NC-ND 4.0](https://creativecommons.org/licenses/by-nc-nd/4.0/)"

在AI Programming时代，不论是AI辅助、还是AI驱动，其对应的自动化、半自动化的整体运行流程必须是可观测的，这是一项基本的要求。

这一点上，应该不会有人有异议。

AI是一个以数据为食的存在，没有数据，它无从分析并改动软件系统中的错误。可观测性在AI时代，是决定系统可靠性与可维护性的基石。

而从人类的角度，AI变成一个完全的黑盒，这也是不可接受的。

我们所讨论的可观测性，体现在三个层面：

1. AI的决策过程
2. AI的执行过程
3. AI生成的代码，运行时的可观测性

下面的讨论，更多的是基于各个部分在整个软件工程中的职责进行的划分。

#### AI决策与执行过程的可观测性

LLM + Agents，在软件工程的层面，我们认为其是来自于当前项目之外的基础设施类服务，参见[Rule 11](https://zhuanlan.zhihu.com/p/2029992166467088525)。

所以，我们将AI决策与执行，在此一并讨论。

本质来讲，这是采用外部(或内部)供应商的技术方案时，对其强制性的规格要求。

LLM + Agents实现提供者，应提供下述信息：

- AI的思维链，我们必要时需要直接检视LLM推理的中间步骤，去反思是Prompt不合适，还是LLM弱智。
- 归因分析，在众多的输入数据中，须标示哪些部分对最终决策影响最大。
- 置信度与不确定性量化，提供模型对其输出结果的把握程度，例如概率与置信区间。
- 编排产生的执行计划的可视性
- 计划执行进度的可视性
- Agents通信协作过程的可视性
- 计划执行输出的日志可审计
- 其它，想到了再补……

#### 软件系统运行时的可观测性

”***我不懂任何编程知识，我讲我要什么，AI生成，然后我把错误扔给AI让它自己改，一遍遍来，直到可以运行***“

每次，看到网络上的小白与人自信满满地争辩”**程序员已死**“时，”水过无痕“的道心就会有一点波澜，会有一点要点一下”反对“的冲动。

这是崽卖爷田的无知、愚蠢、傲慢，并且还不自知。

好，骂完了，心里舒服了，😁

小白可以口嗨，是因为当前已存在的大量类库，可以输出错误日志。

而这些类库可以输出日志，是因为在手工编程时期，保持”可观测性“，已经是我们程序员受训后，形成的一个下意识的习惯性行为。

如果一个方法、类、模块、系统，不再输出日志，嗯，结果太美丽，捂脸不敢看。

在DevOps体系、以及基于云原生的微服务技术体系中，链路与服务的可观测性，是一个规范级的强制要求。

其落地标准，就是OTel (OpenTelemetry)。

在传统人工开发的时代，OTel，已是代表了我们的极致努力。

而在面向AI Programming时，这一切，却还有改进的空间。

我们期望的、面向 AI Programming的Tracing系统的特征，在[《面向AI Programming的服务调用》](https://zhuanlan.zhihu.com/p/2027763667572208559)，及[[Rule 12: 使用透明RPC微服务架构简化需求定义、及代码生成](https://zhuanlan.zhihu.com/p/2030346426811723907)中进行了相关的描述，简单总结下：

1. 不采样，默认关闭Tracing
2. 以一次调用为粒度，动态开启通信-服务链路跟踪
3. 实时可观测性，调用实时透明性
4. Timeline模型。由Action Event构成Timeline，Timeline支持分支、合并等结构，每个Event是某个时刻的系统状态切片，可注入调用参数、当前系统状态等信息
5. 不需要自动织入，AI会自动写入相关代码

基于Timeline模型实现的可观测性，会为AI Agent提供充足的数据，从而支持AI Agent高效进行故障分析及自动排障。

##### 通信与服务调用链路的可观测性

简单来讲，这是**服务框架层**必须原生支持的一个Feature，在面向AI Programming的软件工程实践中不可或缺。

现有日志系统、OTel规范已经提供了一定程度上的支持。

而在面向AI的Programming时，我们需要的更多。

”***在一次业务中，跨多个微服务调用时，支持对整体通信链路、整体服务调用栈完整追踪，以一次调用为粒度，动态开启，详细到一次方法调用的参数级别***“

这是我们开启”**面向AI Programming的透明RPC框架**“讨论，以及着手去实现的众多原因中，最重要的一个最初动力。

缺失了这个Feature，AI基于数据分析进行自动排障的能力将大打折扣。

我们开启了另外一个专栏[KreeX: 透明RPC微服务架构](https://www.zhihu.com/column/c_2027391524585969485)，进行单独讨论。

因为，它很重要，也很复杂，值得一个专栏。

##### 业务逻辑执行的可观测性

在古法编程时代，这个事情很麻烦。

我们会有各种类似于自动织入的解决方案，避免手工编码，这虽然带来了信息可能缺失的问题，但却是在工程成本约束下一个不得不做的折中。

在AI时代，我们过去受到的诸多约束被解除了。

让AI逐行Coding，Who cares？

```javascript
ctx.traceEnabled && tracer.trace('before call save({name: John})', ...args)
```

简单、直接，也无性能问题。
