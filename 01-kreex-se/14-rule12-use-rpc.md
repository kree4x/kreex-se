5. 服务治理原生

   - ……

一个极简，逻辑缜密，雄伟的蓝图，就在我们眼前浮现。

1. Rule 3，使用改良的UML静态图、活动图，辅以流程图、算法伪代码，此种结构化语言，可以驱动AI以可接受的准确度可靠处理。
2. Rule 4，将需求拆分为小模块，每个小模块是一个微服务的粒度，多个小模块组合为一个大的模块，直至搭建整个系统
3. 微服务架构是天选之子，一个模块就是一个微服务
4. 强类型约束的IDL：UML描述的类、方法、参数、返回值，是一个强类型约束的IDL
5. 服务交互过程结构化、语义化：UML的活动图描述了微服务、本地、远程方法间的交互过程
6. 改良UML，为其加入事务、远程方法标志等元素
7. 透明方法调用，抹去本地调用和远程调用差别
8. 服务治理原生，由框架来处理分布式远程服务与本地调用固有的差异性
9. Schema In AI， AI持有UML描述的强类型IDL，Service代码由AI动态生成
10. No Schema In Service，服务层面不需要定义Schema，AI在全局持有
11. 源码即可执行代码，脚本语言具有更大的优势
12. 通信与服务无关，服务与网络分离，剥离AI生成代码时对网络通信复杂处理
13. 运行时类型安全性：AI可以依据UML所定义的IDL在生成代码时注入强制性的语法、逻辑校验。
14. 运行时事务管理：AI根据改良UML加入的事务标志，生成事务控制、失败回滚代码
15. 以一次调用为粒度的实时观测性：Tracing不采样，按需开启
16. Tracing数据是由Action Event构成的带分支、合并的混合多时间线Timeline，每个Event是某个时刻的系统状态切片。
17. Tracing Timeline，它为AI提供的是服务调用全栈分析的数据、参数状态，绝非日志，绝非日志，绝非日志。重要的事情说三遍。
18. 动态服务 + No Schema + 脚本语言，可以使AI以我思故我在的方式编程、Mock测试、集成测试。
19. Rule 9，架构师，需求之后，Coding、DevOps的全程自动化

#### 极大的便利性

- 与LLM的强类型思维认知对齐
  - LLM本身基于概率生成文本，但UML强类型接口为它提供了确定性约束
  - UML作为IDL，作为代码生成“契约”使LLM的代码生成行为可预测、可验证
  - 每个RPC调用都有明确的输入输出类型，减少LLM的猜测空间
- 实现上下文的高效利用，LLM上下文中的知识结构恰好匹配RPC架构
  - 上下文包含：
    - UML动态图描述的服务拓扑图 (Service A → Service B → Service C)
    - UML静态结构描述IDL定义文件 
    - 框架规范 (超时配置、重试策略、熔断规则)
    - 代码生成模板
  - LLM只需“填空”生成业务逻辑
    - LLM不需要理解网络通信的复杂性，只需按模板生成
    - 分布式系统的“难题”被框架吸收，LLM处理“简单题”
    - LLM可专注于单个服务的业务逻辑完整性
- 生成的微服务代码的可组合性
  - 每个服务是独立的代码生成单元
  - 接口不变的情况下，可独立修改和重新生成单个服务
- 类型安全与代码生成的高度确定性
- 设计即代码，需求即代码
- 具备上述特征的透明RPC框架，带来架构治理的自动化。AI 生成的代码自带可观测性和容错能力。架构规范通过 UML 强制执行，而非靠人工 Code Review。



#### 这是一条捷径

AI基于类UML定义的强类型IDL，采用动态语言、无Schema的透明RPC模式进行代码生成、构建与测试。

再结合由ActionEvent组成的Timeline所提供的全链路动态跟踪能力，进行错误分析、修复与迭代演进。

这构成了一条极简的便捷之路。

在面向AI Programming范式下，这条捷径所能带来的效率提升，若转而采用其他技术体系，是否可简单实现尚在两可之间，但是难免会引入更多的复杂性，带来更大的成本和不确定性风险。
