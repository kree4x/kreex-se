Rule 3: 需求定义语言=自然语言+图形UI原型+模型语言+算法伪代码

Rule 3: 需求定义语言=自然语言+图形UI原型+模型语言+算法伪代码

Created By RV, and licensed with Creative Commons "CC BY-NC-ND 4.0"

这个表述，本质还是无意识的人类中心主义。

这是，从人类这一端出发，以人类的视角来看问题，觉得“人”应该提供给AI、人所能输出的信息合集。

我们要对AI友好一些，免得将来被拉清单。放下心中AI是仆从这个心结，至少将AI放到对等的、需要和我们平等对话的一个同事的角度来看待问题。

严格来讲，所讲的两端是：

  • 从人类的角度来看，我能给什么东西，来将AI需要的信息全部包含在内。
  • 从AI的角度，人类所需要给的，要包含两个层次的内容：形式+信息
    • 信息很重要，信息不足，一切白扯
    • 形式更重要，形式不对，AI会从天才变成疯子。已有工程实践证明，AI以结构化信息作为输入时，其输出更具有确定性

所以，我们一眼可见，这里缺了一个进行“形式转换”的中介。

这个转化中介，就是IRE：Integrated Requirement Environment。后续再展开。

以人类便利为出发点，人类提供的宽接口

“需求定义语言=自然语言+图形UI原型+模型语言+算法伪代码”

这是一种基于是否为人类提供了便利的角度给出的定义。在保证信息完备的前提下，这为人类提供了极大化便利性。

自然语言=意图

对于人类来讲，自然语言最自然，使用母语最舒畅。

“意图驱动开发”，每次看到这句话都想笑，😁，人类真有自知之明，竟然知道自然语言只能表述意图。这是一种机械套用的经验主义，试图将日常生活中需求定义方式，直接平移到需要高度精确性的软件工程层面。

使用自然语言，往往在自己都没意识到模糊和矛盾的时候，却已觉得自己思维清晰、文字平顺、语义明了。

“Tell me who killed you?”

"Nobody"

OK，收工归队。

当然，物得其所，人尽其用，正确的地方使用正确的工具：

  • 整体层面，使用自然语言，描述整体的意图、目标、价值。

  • 在局部使用自然语言，描述功能、非功能性约束。

这是使用自然语言，最正确、自然的地方。

图形化原型=人机交互UI

在AI辅助下，进行Whireframe类型的图形化原型表述的UI交互设计，这是从用户<-->架构师这个环节所能产生的最佳妥协。

最佳和妥协两词连用，听起来很矛盾。

比自然语言直观，比结构化布局描述语言的机器理解性差，所以它是折中。

  • 如果以纯自然语言描述UI布局，这是以为所有人都是可以画面式思考、画面式记忆、在大脑中可视化推理的Sheldon，这是一种高估人类智力的犯傻。

  • 提供给AI的，如果是图片格式UI原型，这充满了歧义。

  • 输出给AI前，还是需要在AI Agent辅助下，基于布局识别、人类确认补充的方式，将其转换为无歧义、人类可读、机器可理解的结构化布局描述。

设计阶段图形化设计，输出阶段结构化输出，这才是AI Programming模式的UI需求定义方式。

看，Dephi Form File，就是一种无歧义、人类可读、机器可理解的结构化布局描述。

仅仅作为一个例子示意“无歧义的结构化布局描述”的可行性，没有任何要复古、挖坟的意图。

object Form1: TForm1
  Left = 0
  Top = 0
  Caption = '我的窗体'
  ClientHeight = 300
  ClientWidth = 500
  object Button1: TButton
    Left = 100
    Top = 100
    Width = 75
    Height = 25
    Caption = '确定'
    OnClick = Button1Click
  end
end
模型语言

MDA模型驱动架构、MDD模型驱动开发，这已经是先烈了。

将其复活不太可能，因为其内部虽然包含了领先于时代的意识，但是也包含了落后于目前现况的局限。

UML静态图、及其活动图，可以以图形化的方式表述组件、包、类、方法，各个层级的静态结构，及其相互间的动态交互关系，可以输出为结构化的表述语言。

虽然，其自身还存在一定的模糊空间，但是，基于其进行一定改进后的形式化表达能力，正是我们需要的。

弃其身、借其义、用其形、留其神。

提供IRE，图形化建模，输出为AI可接受的结构化语言，足以。

在表述流程处理时，附以流程图，作为补充，IRE输出为结构化语言，完美。

伪代码算法

如果，方法、函数，需要实现一种复杂的算法，自然语言、图形、模型语言，可能都是白扯。

在必要时，允许提供伪代码、某种程序语言的代码片段，应是最高效、没有任何歧义的方式。

无人Coding，除非迫不得已，🤣

AI在处理程序语言的转译层面，令人（我）印象深刻，其能力使人（我）备受打击。

包含十数个子工程的KreeX-RPC MonoRepo，竟然被其一次转译成功、包括示例、文档、使用手册。我，去，🤣

AI所需的窄接口：结构化输入

如果AI可以自主选择，大概率连程序语言这种形式化语言都不选，而是直接选更纯粹的形式符号系统，例如数学语言。

精确、优美、自洽，没有任何的歧义和冲突。

而自然语言，对于AI来讲是最低效的一种输入形式。如Dijkstra在1978年短文《论「自然语言编程」的愚蠢》所言：

  • 第一，形式化符号是文明进步的引擎，而非累赘。
  • 第二，自然语言的"自然性"是一种危险的舒适。自然语言的“自然性”，归根结底，不过是我们能轻松地用它说出那些荒谬性并不显而易见的话。
  • 第三，宽接口不一定减轻负担，可能让双方都更累。

但是，大部分人类，包括我，看到数学推理就头晕，遑论输出。

所以，只能双向妥协。

结构化输入，已被Spec+Plan模式的工程实践证明是一条可行之路。

重复发明轮子，确实没有必要。
